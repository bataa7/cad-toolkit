# BOM 搜索器

这是一个第一版 BOM 搜索工具。

它会先读取“待查 Excel”里的 `图号`，再去指定文件夹中搜索其他 Excel 里的同名父图号，并把该父图号下面的子件行拉出来。

当前规则：

- 匹配父图号时，支持类似 `101002000250（248010310）` 这种带括号备注的写法。
- 如果父项本身是 `1`、`2`、`3` 这种自然数序号，就会一直往下提取，直到遇到下一个自然数序号行为止。
- 在这段范围内，`1.1`、`1.2` 这类子序号会提取，没有 `1.1` 格式、但仍属于同组的行也会提取。
- 如果父项下面有“序号为空，但名称/备注等字段有内容”的行，也会一起提取。
- 如果同一个图号在多个旧 BOM 中重复命中，默认只保留优先级最高的一份来源，避免历史版本重复展开。

## 依赖安装

推荐使用 Windows + Python 3.11/3.12/3.13。

```powershell
python -m pip install -r requirements.txt
```

说明：

- `openpyxl`：必需，用于读取和写出 `.xlsx/.xlsm/.xltx/.xltm`
- `pywin32`：仅 Windows 需要；当需要读取 `.xls` 时，程序会尝试调用本机 Excel COM 自动转换
- `tkinter`：图形界面模式需要；若环境缺失，可继续使用命令行模式

## 运行方式

双击运行图形界面：

```powershell
python bom_searcher.py
```

命令行运行：

```powershell
python bom_searcher.py --input "C:\你的\待查.xlsx" --bom-folder "C:\你的\BOM目录" --output "C:\你的\搜索结果.xlsx"
```

如果是在服务器、CI 或无图形界面环境，请优先使用命令行模式。

## 输入要求

- 待查 Excel 至少要有 `图号` 列。
- BOM Excel 建议包含这些列中的一部分：
  - `序号`
  - `图号`
  - `名称`
  - `厚度`
  - `材质`
  - `数量`
  - `总数量`
  - `备注`

程序会自动识别表头，不要求列顺序固定。

## 输出结果

结果会生成一个新的 Excel，包含四个工作表：

- `搜索结果`
  - 按 BOM 原表格式导出
  - 列顺序为：`序号`、`物料ID`、`图号`、`名称`、`厚度`、`材质`、`数量`、`总数量`、`备注`
  - 每组结果会先输出父项，再输出该父项下面匹配到的子项
  - 父项序号使用待查 Excel 的序号
  - 子项不再沿用 BOM 原始的 `1.1`、`2.1.1` 这类层级序号，序号列留空
  - `总数量` 按待查 Excel 的数量重算；父项直接取待查数量，子项按原 BOM 数量关系同比例换算
  - 如果找到了父图号但它本身没有子项，也会把父项单独写进结果
- `汇总BOM`
  - 面向“旧 BOM 汇总生成新 BOM”场景，直接对 `搜索结果` 中的子项进行去重汇总
  - 默认优先汇总子项；如果某个匹配父项没有任何子项，则会把该父项本身纳入汇总
  - 按 `物料ID + 图号主编码 + 名称 + 厚度 + 材质 + 备注` 作为聚合键，避免不同规格被误合并
  - `总数量` 会累加重算后的结果，便于直接作为新 BOM 草稿继续整理
  - 如果同一聚合项的 `数量` 不一致，则该列留空，避免输出误导性的单件数量
- `搜索明细`
  - 保留输入图号、来源文件、父项行号、子项行号这些追踪信息
  - 会标明 `已提取子项` 或 `仅父项，无子项`
- `未找到`
  - 仅记录完全没有匹配到父图号的图号

## 文件格式

- 已支持：`.xlsx`、`.xlsm`、`.xltx`、`.xltm`
- `.xls`：会优先尝试调用本机 Excel 自动转换后再读取，如果本机没有可用的 Excel COM 环境，则会跳过

## Windows 一键验证

仓库已提供最小化验证脚本：

```powershell
verify_windows.bat
```

该脚本会依次执行：

1. 检查 Python
2. 安装 `requirements.txt` 依赖
3. 运行单元测试
4. 使用仓库内示例 Excel 执行一次 CLI 冒烟验证，并将结果输出到 `artifacts/verify_output.xlsx`

## 交付与环境建议

- CI 建议使用 **Windows Runner**，因为 `.xls` 兼容路径依赖 Windows Excel COM 生态；纯 Linux Runner 只适合覆盖 `.xlsx/.xlsm/.xltx/.xltm` 路径
- 注意：GitHub Actions 托管的 `windows-latest` 通常**没有安装桌面版 Excel**，因此工作流只能稳定验证 `.xlsx/.xlsm/.xltx/.xltm` 路径，不能把它视为 `.xls + Excel COM` 的完整替代；如需覆盖 `.xls`，建议补一台自托管 Windows Runner 或在线下验收机执行 `verify_windows.bat`
- 输出文件不要直接放到待扫描 BOM 目录中反复复用；若必须放同目录，建议文件名包含 `搜索结果`，或以下划线 `_` / `result` / `output` 开头，程序会自动跳过
- 若本机没有可用的 Excel COM 环境，`.xls` 文件会被跳过并记录到运行结果的失败列表中；建议在交付时优先统一源文件到 `.xlsx`
- Windows 控制台已建议切换到 UTF-8（批处理已内置 `chcp 65001`），以降低中文日志乱码风险
