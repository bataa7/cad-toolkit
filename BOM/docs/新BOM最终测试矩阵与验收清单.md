## 2. 当前实现与验收口径说明

### 2.1 当前代码已实现的能力
- 支持 **CLI** 与 **Tkinter GUI** 两种入口。
- 支持读取待查 Excel、递归扫描 BOM 文件夹、匹配父图号、提取子项、重算数量。
- 当前输出 Excel 已包含 4 个工作表：
  1. `搜索结果`
  2. `汇总BOM`
  3. `搜索明细`
  4. `未找到`
- 已支持 `.xlsx/.xlsm/.xltx/.xltm`，`.xls` 依赖 Windows + Excel COM。

### 2.2 当前代码仍存在的实现边界/风险
以下内容必须在最终验收时重点关注，并区分“目标态”与“当前代码现状”：
1. **异常报告**：文档目标态要求存在 `异常报告`，但当前代码尚未导出该工作表。
2. **说明行误入汇总BOM**：当前 `aggregate_groups()` 会把无图号的工艺说明/备注行也可能纳入 `汇总BOM`。
3. **多来源重复命中**：同一父图号在多个 BOM 文件中出现时，当前实现会全部纳入，不具备版本优先级裁决。
4. **完整图号 vs 主图号口径**：当前默认按 `drawing_base` 匹配与聚合，括号尾码默认忽略。
5. **GUI 文案/状态提示**：文档目标态中的部分 GUI 状态、页签与成功摘要尚未完全体现在当前代码里。

> 结论：最终测试矩阵分为两层：
> - **P0 当前交付必须验证**：围绕当前已实现能力与高风险边界。
> - **P1 目标态验收项**：围绕文档已定义但代码仍待补齐的能力，如异常报告、冲突显式暴露、GUI 结果预览增强等。

## 3. 测试样例数据准备建议

建议至少准备以下 6 组标准样例，覆盖主路径与高风险边界：

### 样例 A：标准成功样例
- 任务单含有效图号、名称、数量
- BOM 库存在父项并可展开 2~5 个子项
- 含正常数量换算
- 用于验证主流程、导出结构、汇总BOM生成

### 样例 B：仅父项无子项样例
- 命中父项，但不存在可提取子项
- 用于验证 parent fallback 逻辑

### 样例 C：完全未命中样例
- 任务单图号在 BOM 库中不存在
- 用于验证 `未找到`、空结果与失败区分

### 样例 D：多来源重复命中样例
- 同一父图号在 2 个旧 BOM 中都命中
- 可含完全相同结构，或仅来源不同
- 用于验证去重、追溯、冲突与重复来源风险

### 样例 E：字段冲突样例
- 同一物料在不同来源中名称/材质/厚度/备注不同
- 用于验证冲突拆分、异常暴露或待确认逻辑

### 样例 F：异常/脏数据样例
- BOM 文件损坏或无法读取
- `.xls` 无法转换
- 数量列为空、非数字、0、格式异常
- 含说明行/空序号行/非标准子序号行

## 4. 最终测试矩阵

以下矩阵按：**功能路径、异常场景、边界数据、导出正确性、CLI/GUI、环境兼容、交付前检查** 组织。

### 4.1 功能路径测试矩阵

| 编号 | 场景 | 输入条件 | 验证步骤 | 预期结果 | 优先级 |
|---|---|---|---|---|---|
| F-01 | CLI 标准成功路径 | 样例A | CLI 执行 `python bom_searcher.py --input ... --bom-folder ... --output ...` | 正常生成输出文件；`搜索结果/汇总BOM/搜索明细/未找到` 存在 | P0 |
| F-02 | GUI 标准成功路径 | 样例A | GUI 选择待查 Excel、BOM 文件夹、输出路径后执行 | 不报错；执行完成；输出文件存在 | P0 |
| F-03 | 父项+多个子项展开 | 样例A | 执行后检查 `搜索结果` 与 `搜索明细` | 父项 1 行 + 子项 N 行；父项在前，子项在后 | P0 |
| F-04 | 非标准子序号提取 | 子项序号为 `A/B` 或空序号说明行 | 执行后检查 `搜索结果` | 非标准子行可被提取到明细/搜索结果 | P0 |
| F-05 | 仅父项无子项 | 样例B | 执行后检查 `搜索结果`、`搜索明细`、`汇总BOM` | 父项保留；明细状态为 `仅父项，无子项`；可回退进入汇总 | P0 |
| F-06 | 多输入图号批量处理 | 待查 Excel 含多个图号 | 执行后核对统计 | 全部输入项均有对应输出归宿：结果/未找到 | P0 |
| F-07 | 汇总BOM生成 | 样例A/D/E | 检查 `汇总BOM` | 产生去重汇总视图；行数不应简单等于搜索结果行数 | P0 |
| F-08 | 追溯明细保留 | 样例A/D/E | 检查 `搜索明细` | 保留输入工作表、来源文件、父/子行号、状态 | P0 |
| F-09 | 未命中输出 | 样例C | 检查 `未找到` | 完全未命中的输入图号全部进入 `未找到` | P0 |
| F-10 | 异常报告输出（目标态） | 样例D/E/F | 检查 `异常报告` | 多来源冲突/数量异常/跳过文件记录可见 | P1 |

### 4.2 去重、数量累计、冲突拆分、未命中核心矩阵

| 编号 | 场景 | 核心断言 | 当前验收口径 |
|---|---|---|---|
| C-01 | 去重：相同物料跨来源重复 | `汇总BOM` 同物料仅 1 行，追溯层保留多来源 | P0，必须验证 |
| C-02 | 去重：图号格式差异但主编码相同 | `101...（A）` 与 `101...` 按 `drawing_base` 归并 | P0，按当前实现 |
| C-03 | 数量累计：同一物料在多来源/多输入出现 | `汇总BOM.总数量 = 所有有效来源重算后总和` | P0 |
| C-04 | 数量累计：小数数量 | 累加与比例换算准确；导出无明显精度错误 | P0 |
| C-05 | 冲突拆分：材质/厚度不同 | 不应被误合并；应拆分为多行或进入异常 | P0 |
| C-06 | 冲突拆分：名称/备注冲突 | 不允许静默覆盖；需可追踪 | P0 |
| C-07 | 未命中 | 所有未匹配输入仅进入 `未找到`，不应混入汇总 | P0 |
| C-08 | 数量无法换算 | 不得输出误导性数量；目标态进入异常报告 | P1 |

## 5. 手工回归步骤（推荐执行顺序）

### 5.1 CLI 主线回归
1. 安装依赖：
   ```powershell
   python -m pip install -r requirements.txt
   ```
2. 执行自动化测试：
   ```powershell
   python -m unittest discover -s tests -v
   ```
3. 使用标准成功样例执行 CLI：
   ```powershell
   python bom_searcher.py --input "C:\路径\任务单.xlsx" --bom-folder "C:\路径\BOM库" --output "C:\路径\输出.xlsx"
   ```
4. 打开输出文件，逐项核对：
   - `搜索结果`
   - `汇总BOM`
   - `搜索明细`
   - `未找到`
5. 用多来源重复命中样例重复步骤 3~4，重点核对去重与追溯。
6. 用异常/脏数据样例重复步骤 3~4，重点核对 skipped_files、数量降级、异常暴露。

### 5.2 GUI 主线回归
1. 执行：
   ```powershell
   python bom_searcher.py
   ```
2. 选择待查 Excel。
3. 选择 BOM 文件夹。
4. 确认/修改输出文件路径。
5. 点击开始执行。
6. 观察：
   - 执行按钮是否禁用
   - 日志是否持续刷新
   - 界面是否卡死
7. 执行完成后核对：
   - 成功/失败弹窗内容
   - 跳过文件日志
   - 输出 Excel 是否生成
8. 用空结果、异常样例、重复命中样例各回归一次。

### 5.3 Windows 一键回归
执行：
```powershell
verify_windows.bat
```
检查：
- Python 检查通过
- 依赖安装通过
- 单元测试通过
- CLI 冒烟输出 `artifacts/verify_output.xlsx`

2. 多来源重复命中去重/显式保留策略
3. 小数数量与累计精度
4. 字段冲突拆分
5. `.xls` 跳过与 skipped_files 记录
6. 输出工作表完整性校验

## 9. 建议的验收结论判定标准

### 可判定为“通过”
同时满足以下条件：
- P0 阻断项全部通过；
- 标准成功样例、空结果样例、重复命中样例、异常样例均完成验收；
- 汇总BOM不再被说明行污染；
- 业务已确认当前多来源命中与图号口径可接受。

### 可判定为“条件通过”
- 主流程可用；
- 但 `异常报告`、GUI 预览增强、冲突显式拆分等仍处于补强阶段；
- 同时业务已接受通过人工复核弥补。

### 不可上线/不可交付
任一情况出现即建议阻断：
- 汇总BOM数量计算错误；
- 说明行进入汇总BOM；
- 多来源重复导致主汇总明显重复且无提示；
- 输出结构与 README/交付说明不一致；
- CLI 或 GUI 任一主入口不可用；
- 交付环境无法稳定运行依赖。

## 10. QA结论

当前项目已经具备“搜索 + 汇总草稿导出”的可运行基础，但最终验收必须把重点放在以下四类高风险能力上：
1. **去重是否正确**
2. **数量累计是否准确**
3. **冲突是否被显式暴露而非静默覆盖**
4. **未命中/跳过文件/异常数据是否可追溯**

建议交付前至少完成：
- 1 次 `verify_windows.bat`
- 1 次 CLI 主线回归
- 1 次 GUI 主线回归
- 4 组标准样例验收（成功/空结果/冲突/异常）
- 说明行过滤与重复来源口径的最终确认
