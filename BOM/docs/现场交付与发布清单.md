# 现场交付与发布清单

本文档用于项目最终现场交付、发布准备、验收执行和交付包核对。目标是让交付人员、实施人员、现场演示人员或项目负责人，在正式交付前后都可以按 checklist 快速确认“带了什么、怎么跑、怎么验、缺什么”。

---

## 1. 交付目标

本项目的现场交付目标应至少满足以下几点：

1. 交付包中包含主程序、依赖说明、运行文档、验证脚本和样例文件
2. 现场机器能够完成普通 `.xlsx` 路径的最小验收
3. 若客户使用 `.xls`，现场机器还应完成 `.xls + Excel COM` 专项验收
4. 演示人员能够按 GUI 或 CLI 路径完成最小演示
5. 交付时能明确说明 CI 覆盖范围与 `.xls` 的环境边界

---

## 2. 交付包应包含的文件

建议现场交付包至少包含以下文件与目录。

### 2.1 程序与脚本

- [ ] `bom_searcher.py`  —— 主程序，支持 GUI + CLI
- [ ] `run_bom_searcher.bat` —— GUI 启动入口
- [ ] `verify_windows.bat` —— Windows 一键验证脚本
- [ ] `requirements.txt` —— 依赖声明

### 2.2 文档入口

- [ ] `README.md` —— 仓库总览与快速启动入口
- [ ] `docs/交付验收与运行手册.md` —— 常规运行/验收手册
- [ ] `docs/Windows + .xls + Excel COM 专项验收前置条件.md` —— `.xls` 专项说明

### 2.3 测试与样例材料

- [ ] `tests/` —— 单元测试目录
- [ ] `测试.xlsx` —— 常规演示/验证样例输入
- [ ] `260118.xlsx` —— 样例 BOM 数据
- [ ] 如现场存在 `.xls` 需求，建议额外准备至少 1~2 份真实 `.xls` 样例

### 2.4 CI / 配置文件

- [ ] `.github/workflows/windows-verify.yml` —— 最小 Windows CI 参考
- [ ] `.gitignore` —— 结果产物与缓存忽略策略

### 2.5 可选但推荐纳入交付包的材料

- [ ] 已验证通过的样例输出文件（例如：`artifacts/verify_output.xlsx`）
- [ ] 客户现场专用演示数据
- [ ] `.xls` 兜底方案说明（如需手工转 `.xlsx`）
- [ ] 现场联系人与问题反馈渠道说明

## 3. README / 文档入口检查

交付前请确认 README 能作为总入口，至少能让现场人员找到以下内容：

- [ ] 如何安装依赖
- [ ] 如何运行 GUI
- [ ] 如何运行 CLI
- [ ] 如何执行 `verify_windows.bat`
- [ ] 交付验收与运行手册入口
- [ ] `.xls/Excel COM` 专项说明入口

### 推荐入口顺序

建议现场使用时按以下顺序查阅：

1. `README.md`
2. `docs/交付验收与运行手册.md`
3. 若涉及 `.xls`，继续看 `docs/Windows + .xls + Excel COM 专项验收前置条件.md`

## 4. 验证脚本与样例输入输出

### 4.1 必备验证脚本

- [ ] `verify_windows.bat` 可以直接运行
- [ ] 批处理执行后能完成依赖安装、单测、CLI 冒烟验证
- [ ] 现场交付人员知道脚本输出位置

### 4.2 样例输入材料

普通 `.xlsx` 验证建议准备：

- [ ] `测试.xlsx`
- [ ] `260118.xlsx`

`.xls` 专项验证建议额外准备：

- [ ] 至少 1 个真实 `.xls` 待查样例
- [ ] 至少 1 个真实 `.xls` BOM 样例（如客户实际会这样使用）
- [ ] 样例文件路径尽量放在本地短路径目录

### 4.3 样例输出材料

交付建议至少准备以下一种输出作为“预期参考”：

- [ ] `artifacts/verify_output.xlsx`
- [ ] 或现场提前生成并核对过的演示输出文件

输出文件应至少包含以下工作表：

- [ ] `搜索结果`
- [ ] `汇总BOM`
- [ ] `搜索明细`
- [ ] `未找到`

## 5. 普通 `.xlsx` 验收所需材料

普通路径是最小交付必须覆盖的路径，建议现场至少准备：

- [ ] Windows 机器
- [ ] Python 运行环境
- [ ] `requirements.txt`
- [ ] `verify_windows.bat`
- [ ] `测试.xlsx`
- [ ] `260118.xlsx`
- [ ] 输出目录（建议 `artifacts/`）

### 推荐执行步骤

- [ ] 执行 `python -m pip install -r requirements.txt`
- [ ] 执行 `verify_windows.bat`
- [ ] 打开 `artifacts/verify_output.xlsx`
- [ ] 抽查 4 个工作表内容

### 普通 `.xlsx` 路径通过标准

- [ ] 单元测试通过
- [ ] CLI 冒烟运行成功
- [ ] 成功生成输出文件
- [ ] 输出结构与内容符合预期

---

## 6. `.xls / Excel COM` 验收所需材料

如果客户现场会使用 `.xls`，仅普通 `.xlsx` 验收是不够的，需要单独准备以下材料。

### 6.1 环境材料

- [ ] Windows 机器
- [ ] Python 环境
- [ ] `requirements.txt`
- [ ] `pywin32` 可用
- [ ] 已安装桌面版 Excel
- [ ] 当前本地登录用户可正常手工打开 Excel

### 6.2 样例材料

- [ ] 至少 1 份真实 `.xls` 样例
- [ ] 如有需要，准备 `.xls` 待查文件与 `.xls` BOM 文件各 1 份以上
- [ ] 如样例文件来自客户现场，提前确认可脱敏演示

### 6.3 文档与说明材料

- [ ] `docs/Windows + .xls + Excel COM 专项验收前置条件.md`
- [ ] 现场人员知道 `.xls` 路径比 `.xlsx` 多依赖 Excel COM
- [ ] 现场人员知道 CI 不能替代 `.xls` 真机验收

### 6.4 推荐执行步骤

- [ ] 执行 `python -m pip install -r requirements.txt`
- [ ] 执行 `python -c "import win32com.client; print('pywin32 ok')"`
- [ ] 手工打开一次 Excel
- [ ] 手工打开至少 1 个真实 `.xls`
- [ ] 使用 CLI 执行 `.xls` 专项验证
- [ ] 检查输出文件、日志和工作表结构

### `.xls / Excel COM` 路径通过标准

- [ ] Excel COM 可用
- [ ] `.xls` 文件未被跳过
- [ ] 输出文件生成成功
- [ ] 输出工作表结构完整
- [ ] 若现场环境复杂，已确认“手工转 `.xlsx`”兜底方案可行

## 7. 现场演示准备项

正式演示前建议至少准备以下内容。

### 7.1 设备与环境

- [ ] 演示机为 Windows
- [ ] 演示机已安装 Python
- [ ] 已预装依赖或可现场联网安装依赖
- [ ] 若演示 `.xls`，演示机已安装桌面版 Excel
- [ ] 演示机已确认可正常打开 Excel
- [ ] 演示机终端可正常显示中文（推荐 UTF-8）

### 7.2 数据与文件

- [ ] 演示数据已放在本地目录
- [ ] 路径不要过深，不放在权限不稳定的网络盘
- [ ] 已准备 GUI 演示用输出路径
- [ ] 已准备 CLI 演示用输出路径
- [ ] 已准备至少一份预期结果文件，便于对照讲解

### 7.3 演示流程建议

#### 演示方案 A：最快路径（推荐）

1. [ ] 先执行 `verify_windows.bat`
2. [ ] 打开 `artifacts/verify_output.xlsx`
3. [ ] 演示四个工作表分别代表什么
4. [ ] 如客户需要，再补 GUI 演示

#### 演示方案 B：GUI 现场操作

1. [ ] 运行 `run_bom_searcher.bat`
2. [ ] 选择待查 Excel
3. [ ] 选择 BOM 文件夹
4. [ ] 选择输出结果路径
5. [ ] 点击“开始搜索”
6. [ ] 打开生成结果，说明 `搜索结果 / 汇总BOM / 搜索明细 / 未找到`

#### 演示方案 C：`.xls` 专项演示

1. [ ] 先说明 `.xls` 依赖 Excel COM
2. [ ] 手工证明 Excel 能正常打开 `.xls`
3. [ ] 运行 `.xls` CLI 命令
4. [ ] 展示输出结果
5. [ ] 若失败，说明兜底方案：先转 `.xlsx` 再运行

## 8. 交付时容易遗漏的检查点

以下是现场最容易遗漏、但会直接影响交付体验的项目。

### 8.1 文件与文档遗漏

- [ ] 忘记带 `requirements.txt`
- [ ] 忘记带 `verify_windows.bat`
- [ ] README 没有文档入口
- [ ] 没带 `.xls` 专项说明文档
- [ ] 没带测试/样例文件

### 8.2 环境遗漏

- [ ] 演示机未安装 Python
- [ ] 演示机未安装 Excel（但客户要求验 `.xls`）
- [ ] `pywin32` 未安装
- [ ] Excel 第一次启动未初始化
- [ ] 现场机器不能联网，导致依赖无法临时安装

### 8.3 数据与路径遗漏

- [ ] 输出文件写回 BOM 目录，导致结果回扫污染
- [ ] 输出路径未提前创建或无写权限
- [ ] 演示数据放在网络盘或权限不稳定目录
- [ ] `.xls` 样例没有提前验证能否手工打开

### 8.4 验收口径遗漏

- [ ] 没有向用户说明 `.xlsx` 与 `.xls` 是两类验收路径
- [ ] 把 GitHub CI 通过误当成 `.xls` 也通过
- [ ] 没有准备 `.xls` 的兜底方案（手工转 `.xlsx`）
- [ ] 没有明确告知输出文件建议写到独立目录

## 11. 推荐交付顺序

现场建议按以下顺序推进：

1. [ ] 先发交付包
2. [ ] 打开 `README.md` 说明入口
3. [ ] 执行 `verify_windows.bat` 完成普通 `.xlsx` 验证
4. [ ] 打开结果文件说明输出结构
5. [ ] 如现场涉及 `.xls`，再执行专项验收
6. [ ] 如现场仍不稳定，说明兜底方案：先转 `.xlsx`

---

**结论**：最终交付不是只交一个程序文件，而是要同时交付“程序 + 依赖 + 文档 + 验证脚本 + 样例 + 验收口径”。只要按本清单逐项核对，普通 `.xlsx` 交付和 `.xls/Excel COM` 增强交付都可以形成清晰、可复现、可演示的现场发布流程。
