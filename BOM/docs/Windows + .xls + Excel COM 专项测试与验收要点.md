## 2. 适用范围与当前实现边界

### 2.1 适用范围
当出现以下任一情况时，必须执行本专项验收：
- 客户现场仍使用 `.xls` 老格式 BOM 文件
- 交付要求中明确写有“必须兼容 `.xls`”
- 现场验收机与开发机不同，需要重新确认 Excel COM 可用性
- `.xlsx` 路径已通过，但 `.xls` 文件在现场出现读取失败、跳过、结果缺失等问题

### 2.2 当前实现边界
根据当前代码实现：
- `.xlsx/.xlsm/.xltx/.xltm` 由 `openpyxl` 直接读取，属于常规稳定路径
- `.xls` 不是直接解析，而是通过 `DispatchEx("Excel.Application")` 调起 **本机 Excel COM**，先另存为临时 `.xlsx`，再继续读取
- `maybe_convert_xls()` 当前在 **BOM 源文件加载路径** 中生效
- `collect_queries()` 当前仍直接使用 `openpyxl.load_workbook(input_file)` 读取待查文件

### 2.3 本次专项验收的关键口径
因此，当前版本应按以下口径验收：
1. **BOM 文件夹中的 `.xls` 文件**：属于本专项重点验证对象
2. **待查输入文件 `input_file`**：建议使用 `.xlsx`，不要把“待查文件为 `.xls`”作为当前版本默认通过项
3. 如业务强制要求“待查文件也必须原生支持 `.xls`”，应单列为 **P0 边界确认项/缺口项**，不能被普通 `.xlsx` 验收结果替代

## 3. 测试前置条件

### 3.1 环境前置条件
专项验收机器建议满足以下条件：
- 操作系统：Windows 10 / Windows 11
- Python：3.11 / 3.12 / 3.13
- 已安装项目依赖：
  ```powershell
  python -m pip install -r requirements.txt
  ```
- Python 可导入 `win32com.client`
- 本机已安装桌面版 Excel，并已完成首次启动初始化
- 当前执行账号为可交互的本地登录用户，而不是受限服务账号

### 3.2 建议专项前置检查
在正式执行前，先完成以下检查：

1. 检查 Python / pywin32：
   ```powershell
   python --version
   python -c "import win32com.client; print('pywin32 ok')"
   ```
2. 手工打开一次 Excel
3. 手工打开至少 1 个真实 `.xls` 文件
4. 确认 Excel 没有阻塞弹窗：许可证、隐私、恢复、加载项、兼容性提示等
5. 先执行一次普通 `.xlsx` 基线验证：
   ```powershell
   verify_windows.bat
   ```

### 3.3 测试数据前置条件
建议准备以下样例：
- 1 份 `.xlsx` 待查文件
- 1 份可正常打开的真实 `.xls` BOM 文件
- 1 份与之内容相同或接近的 `.xlsx` BOM 文件，用于结果对照
- 1 份故意损坏或需修复的 `.xls` 文件，用于失败场景验证
- 1 份混合目录样例：同时包含 `.xls`、`.xlsx`、临时文件 `~$`、旧输出文件

## 4. 与普通 `.xlsx` 路径的差异

| 对比项 | `.xlsx/.xlsm/.xltx/.xltm` 路径 | `.xls + Excel COM` 路径 |
|---|---|---|
| 读取方式 | `openpyxl` 直接读取 | 先调用 Excel COM 转临时 `.xlsx` 再读取 |
| 环境依赖 | Python + openpyxl | Windows + Python + pywin32 + 桌面 Excel + COM |
| CI 可验证性 | 可由常规 Windows CI 稳定覆盖 | GitHub 托管 `windows-latest` 不能视为完整替代 |
| 失败模式 | 主要是文件损坏/表头无法识别 | 额外叠加 COM 创建失败、Excel 首启弹窗、权限/桌面会话问题 |
| 性能特征 | 相对稳定 | 首次/批量 `.xls` 会更慢 |
| 结果稳定性 | 主要取决于数据本身 | 既取决于数据，也取决于本机 Excel 状态 |
| 输入文件边界 | 当前稳定支持 | 当前版本不建议把待查输入文件 `.xls` 作为默认通过项 |

## 5. 关键测试路径

### 5.1 执行原则
- **先跑 `.xlsx` 基线路径，再跑 `.xls` 专项路径**
- **优先使用 CLI** 做专项验收，GUI 仅用于补充用户视角验证
- `.xls` 专项必须至少使用 1 份真实业务样例，不建议只用空白或人工极简样例

### 5.2 专项测试路径矩阵

| 编号 | 场景 | 输入/环境 | 验证步骤 | 预期结果 | 优先级 |
|---|---|---|---|---|---|
| XLS-01 | `.xlsx` 基线先通过 | 普通 Windows 验收机 | 先执行 `verify_windows.bat` | `.xlsx` 主路径通过，排除基础环境问题 | P0 |
| XLS-02 | `.xlsx` 待查 + 单个 `.xls` BOM | Windows + Excel COM 可用 | CLI 执行新BOM生成 | 成功生成输出；`.xls` 内容参与检索与汇总 | P0 |
| XLS-03 | `.xlsx` 待查 + 混合目录（`.xls` + `.xlsx`） | 混合 BOM 文件夹 | CLI 执行并核对结果 | 有效 `.xls/.xlsx` 均被处理；无效文件被跳过并记录 | P0 |
| XLS-04 | 多个 `.xls` BOM，其中 1 个损坏 | 包含坏文件 | CLI 执行并检查日志 | 主流程继续；坏文件进入 skipped_files/跳过记录 | P0 |
| XLS-05 | `.xls` 转换后结果对照 | 同内容 `.xls` 与 `.xlsx` 各一份 | 分别执行，对比输出关键结果 | 在相同数据前提下，核心匹配/汇总结果一致 | P0 |
| XLS-06 | GUI 选择 `.xlsx` 待查 + `.xls` BOM 目录 | GUI 可用 | 人工执行一次 GUI 路径 | GUI 不崩溃；结果文件可生成；日志可见跳过/成功信息 | P1 |
| XLS-07 | 待查输入文件本身为 `.xls` | `.xls` 作为 input_file | 执行 CLI 或 GUI | 当前版本应视为边界项：需预转 `.xlsx` 或列为不通过/待补强 | P0 边界 |
| XLS-08 | 网络盘/长路径/权限目录中的 `.xls` | 非本地短路径 | 执行 CLI | 若失败，应能通过迁移到本地短路径复现/规避 | P1 |

### 5.3 每条路径的核心断言

#### XLS-02 / XLS-03 / XLS-05 必核对
- 输出文件成功生成
- 输出文件至少包含：
  - `搜索结果`
  - `汇总BOM`
  - `搜索明细`
  - `未找到`
- `.xls` 中的有效父项/子项确实被纳入结果，不是被静默跳过
- `汇总BOM` 中数量重算、去重结果与 `.xlsx` 对照样例一致

#### XLS-04 必核对
- 单个坏 `.xls` 不应拖垮全部任务
- 日志或结果摘要应能定位被跳过文件
- 与坏文件无关的 `.xlsx` / 其他 `.xls` 仍应正常参与结果生成

#### XLS-07 必核对
- 当前版本不得把“待查 `.xls` 输入”误判为“已完整支持”
- 如果现场必须支持该路径，应升级为开发缺口或交付边界说明

## 6. 常见失败场景与验收要点

| 编号 | 失败场景 | 典型现象 | 验收判断 | 建议处理 |
|---|---|---|---|---|
| F-COM-01 | 缺少 `pywin32` | `import win32com.client` 失败 | 直接判定 `.xls` 专项不可通过 | 重装依赖并确认 Python 环境一致 |
| F-COM-02 | 未安装桌面 Excel | `.xlsx` 正常，`.xls` 全失败/被跳过 | 不能判定 `.xls` 支持通过 | 安装 Excel 并手工打开初始化 |
| F-COM-03 | Excel 首次启动弹窗阻塞 | COM 调用卡住、失败、无输出 | 高风险阻断 | 手工启动 Excel，清完首次弹窗 |
| F-COM-04 | 受限账号/无桌面会话 | 远程/服务账号下 COM 异常 | 高风险阻断 | 改为本地登录用户验收 |
| F-COM-05 | `.xls` 文件损坏或需修复 | 手工打开先弹修复提示 | 单文件失败可接受，整体失败不可接受 | 先人工修复并另存为 `.xlsx` |
| F-COM-06 | 网络盘/权限/长路径问题 | 某些文件无法打开 | 中高风险 | 迁移到本地短路径复验 |
| F-COM-07 | 输出回扫污染 | 第二次运行结果异常增多 | 不是 COM 问题，但现场高频误判源 | 输出写到独立 `artifacts` 目录 |
| F-COM-08 | 待查文件本身为 `.xls` | 程序起始阶段即失败 | 当前版本边界，不应算已支持 | 待查文件先转 `.xlsx` |

## 7. 风险分级

### 7.1 P0 阻断风险
以下任一出现，若业务要求支持 `.xls`，则不得判定专项验收通过：
1. 机器不是 Windows
2. `pywin32` 不可用
3. 本机没有可用桌面 Excel / Excel COM 无法创建
4. `.xls` 业务样例无法被程序处理，且不是单文件损坏导致
5. 现场必须支持“待查文件为 `.xls`”，但当前版本未明确实现
6. `.xls` 参与后生成结果明显缺项、数量错误、全部被静默跳过

### 7.2 P1 高风险
以下问题不一定阻断整体交付，但必须记录并给出规避措施：
1. Excel 首次启动/恢复/加载项弹窗导致 COM 不稳定
2. 网络盘、权限目录、长路径导致部分 `.xls` 失败
3. 混合目录中部分 `.xls` 成功、部分失败
4. GUI 路径下提示不足，导致用户误以为 `.xls` 已被成功处理

### 7.3 P2 中风险
以下问题可作为条件通过项，但需在交付说明中明确：
1. CI 只能覆盖 `.xlsx` 路径，不能代替 `.xls + COM` 实机验收
2. `.xls` 转换速度明显慢于 `.xlsx`
3. `.xls` 支持依赖本机 Office 状态，跨机器一致性弱于 `.xlsx`

## 8. 无法提供 Excel COM 环境时的替代验收建议

### 8.1 可接受的替代验收前提
只有在以下前提满足时，才可采用替代验收：
- 业务接受“现场统一把 `.xls` 转成 `.xlsx` 再使用”
- 或合同/交付口径未把“原生 `.xls` 读取”列为必须项
- 或本次先验收主功能，`.xls` 兼容作为后续增强项

### 8.2 推荐替代方案

#### 方案 A：人工转 `.xlsx` 后按主矩阵验收
1. 由验收人员手工打开 `.xls`
2. 另存为 `.xlsx`
3. 使用转换后的文件执行《新BOM最终测试矩阵与验收清单》中的标准回归
4. 记录结论：
   - `.xlsx` 主路径通过
   - `.xls` 原生 COM 路径未在现场验证

#### 方案 B：批量预转换后交付
适用于客户现场长期无法提供 Excel COM 环境：
- 在交付前统一把历史 `.xls` 批量转换为 `.xlsx`
- 交付运行手册中明确：生产使用仅保证 `.xlsx` 路径
- `.xls` 作为非默认支持格式，不纳入现场阻断项

#### 方案 C：条件通过 + 风险说明
如主流程全部通过，但现场无法提供 Excel COM，可采用以下口径：
- 本次交付确认 `.xlsx` 路径可用
- `.xls` 原生兼容性因现场缺少 Excel COM 环境，未完成最终实机验证
- 若后续必须启用 `.xls`，需在具备桌面 Excel 的 Windows 机器上补专项验收

### 8.3 不可接受的替代方式
以下情况不能作为“.xls 已通过”的依据：
- 仅看 GitHub Actions `windows-latest` 结果
- 仅验证 `.xlsx` 样例后口头推断 `.xls` 也可用
- 仅确认安装了 Office，但未实际打开 Excel / `.xls`
- 未用真实 `.xls` 文件跑过 CLI 或 GUI，就直接签收“.xls 支持通过”

## 9. 最终专项验收清单

### 9.1 通过前必须满足
- [ ] 当前机器为 Windows
- [ ] `python -m pip install -r requirements.txt` 已执行
- [ ] `python -c "import win32com.client; print('pywin32 ok')"` 通过
- [ ] 已手工打开 Excel 且无阻塞弹窗
- [ ] 已手工打开至少 1 个真实 `.xls`
- [ ] 已完成 `.xlsx` 基线验证（`verify_windows.bat`）
- [ ] 已完成 `.xlsx` 待查 + `.xls` BOM 的 CLI 专项验证
- [ ] 已检查输出文件结构与关键结果
- [ ] 已核对坏 `.xls` 不会拖垮全部流程
- [ ] 已确认待查 `.xls` 输入是否在本次交付范围内

### 9.2 可判定为“专项通过”
同时满足以下条件：
- `.xlsx` 基线路径通过
- 真实 `.xls` BOM 文件在 Windows + Excel COM 环境下可被处理
- 输出结果与 `.xlsx` 对照路径在关键业务结果上保持一致
- 单个坏 `.xls` 不影响整体流程
- 验收结论明确记录了 `.xls` 的支持边界

### 9.3 只能判定为“条件通过”
- `.xlsx` 路径通过
- 但现场无 Excel COM，只能采用“先转 `.xlsx` 再运行”的替代方案
- 同时业务/交付方书面接受该口径

### 9.4 必须判定为“不通过/待补强”
- 现场明确要求原生 `.xls` 支持
- 但没有真实 Windows + Excel COM 机器完成实测
- 或实测中 `.xls` 全部失败/核心结果错误
- 或待查 `.xls` 输入被要求支持但当前版本未落地

## 10. QA建议结论

对于最终交付，建议统一采用以下表述：

1. **`.xlsx` 是默认稳定路径**，可通过常规自动化和 Windows 一键验证覆盖
2. **`.xls` 是增强兼容路径**，其通过结论必须以真实 Windows + 桌面 Excel + COM 实测为准
3. **当前专项重点应放在 BOM 源文件 `.xls` 兼容**，而不是默认承诺“所有输入位置的 `.xls` 都已完全支持”
4. **如现场无法提供 Excel COM 环境，应明确采用“预转 `.xlsx`”的替代验收口径**，不能把它表述为“.xls 原生路径已通过”
