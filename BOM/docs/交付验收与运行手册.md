# 交付验收与运行手册

本文档面向最终交付、现场演示、验收测试与后续维护人员，目标是让他人在不依赖项目开发背景的情况下，能够完成环境准备、程序启动、自动验证、结果检查与问题排查。

---

## 1. 适用范围

当前项目为 **Windows 优先** 的 Python Excel 工具，核心用途是：

- 读取“待查 Excel”中的图号
- 到指定 BOM 文件夹中搜索匹配父图号
- 抽取对应子件明细
- 生成新的结果文件与汇总 BOM

当前仓库已具备以下最小交付保障：

- 显式依赖清单：`requirements.txt`
- Windows 一键验证脚本：`verify_windows.bat`
- 最小 Windows CI：`.github/workflows/windows-verify.yml`
- GUI 启动脚本：`run_bom_searcher.bat`
- 结果产物忽略策略：`.gitignore`

## 2. 环境准备

### 2.1 推荐运行环境

推荐使用以下环境：

- 操作系统：**Windows 10 / Windows 11**
- Python：**3.11 / 3.12 / 3.13**
- 终端：Windows Terminal、PowerShell、CMD 均可
- Excel：
  - 处理 `.xlsx / .xlsm / .xltx / .xltm` 时，不强制要求安装桌面版 Excel
  - 若需要处理 `.xls`，**建议安装本机桌面版 Excel**，并确保可用 Excel COM 组件

### 2.2 必备/可选组件

| 组件 | 是否必需 | 用途 |
|---|---|---|
| Python | 必需 | 运行程序与验证脚本 |
| openpyxl | 必需 | 读取/写出 Excel 新格式 |
| pywin32 | Windows 下建议安装 | `.xls` 场景调用 Excel COM |
| tkinter | GUI 模式需要 | 启动图形界面 |
| 桌面版 Excel | `.xls` 场景建议具备 | 将 `.xls` 转换为 `.xlsx` 后再读取 |

### 2.3 环境自检命令

在项目根目录执行：

```powershell
python --version
python -m pip --version
```

如需确认 GUI 与 `.xls` 支持，可进一步执行：

```powershell
python -c "import tkinter; print('tkinter ok')"
python -c "import win32com.client; print('pywin32 ok')"
```

## 4. 启动方式

## 4.1 GUI 启动方式

适合业务同事或现场演示人员。

方法一：双击或运行批处理：

```powershell
run_bom_searcher.bat
```

方法二：直接运行：

```powershell
python bom_searcher.py
```

GUI 启动后，需要选择：

1. 待查 Excel
2. BOM 文件夹
3. 输出结果文件位置

然后点击“开始搜索”。

> 注意：GUI 模式依赖 `tkinter`。如果当前 Python 环境缺少 tkinter，请改用 CLI 模式运行。

## 4.2 CLI 启动方式

适合自动化验证、CI、服务器环境或无图形界面环境。

```powershell
python bom_searcher.py --input "C:\你的\待查.xlsx" --bom-folder "C:\你的\BOM目录" --output "C:\你的\搜索结果.xlsx"
```

示例：

```powershell
python bom_searcher.py --input "C:\Users\1\Desktop\BOM\测试.xlsx" --bom-folder "C:\Users\1\Desktop\BOM" --output "C:\Users\1\Desktop\BOM\artifacts\verify_output.xlsx"
```

## 5. 推荐的一键验证方式

项目已提供 Windows 一键验证脚本：

```powershell
verify_windows.bat
```

该脚本按以下顺序执行：

1. 检查 Python 是否可用
2. 安装 `requirements.txt` 中声明的依赖
3. 执行单元测试：`python -m unittest discover -s tests -v`
4. 使用仓库示例数据执行一次 CLI 冒烟验证
5. 输出验证产物到：`artifacts\verify_output.xlsx`

### 5.1 一键验证通过的判定标准

以下条件同时满足，可视为最小交付链路通过：

- 单元测试全部通过
- CLI 冒烟执行完成
- 生成 `artifacts\verify_output.xlsx`
- 输出文件可正常打开，且包含预期工作表

## 6. 推荐的最小演示/验收流程

这是推荐给 Leader、业务同事或最终验收人的最小验收路径。

### 步骤一：准备环境

在 Windows 机器上：

```powershell
python -m pip install -r requirements.txt
```

### 步骤二：执行自动验证

```powershell
verify_windows.bat
```

### 步骤三：检查验证产物

打开以下文件：

```text
artifacts\verify_output.xlsx
```

检查是否包含以下工作表：

- `搜索结果`
- `汇总BOM`
- `搜索明细`
- `未找到`

### 步骤四：人工抽样确认

建议至少确认以下内容：

1. `搜索结果` 中是否先输出父项，再输出对应子项
2. `汇总BOM` 中是否按图号/规格正确汇总
3. `搜索明细` 中是否能追踪来源文件、来源工作表、父行号、子行号
4. `未找到` 中是否列出了未命中的输入图号

### 步骤五：GUI 演示（可选但推荐）

执行：

```powershell
run_bom_searcher.bat
```

然后在界面中手工选择：

- 待查 Excel：`测试.xlsx`
- BOM 文件夹：项目根目录或实际 BOM 文件夹
- 输出文件：建议写到 `artifacts\demo_output.xlsx`

点击“开始搜索”，观察日志与结果文件生成情况。

### 步骤六：`.xls` 特殊验收（如项目实际会用到）

如果客户现场存在 `.xls` 文件，建议在**安装了桌面版 Excel 的 Windows 机器**上，额外使用一份真实 `.xls` 样例执行一次人工验收。GitHub 托管 CI 不能替代这一步。

专项前置条件、失败模式与用户排查建议，详见：

- `Windows + .xls + Excel COM 专项验收前置条件.md`

## 7. 输出产物说明

### 7.1 默认/常见输出位置

| 场景 | 产物位置 |
|---|---|
| 一键验证 | `artifacts\verify_output.xlsx` |
| GUI 手动运行 | 由用户自行选择 |
| CLI 手动运行 | 由 `--output` 指定 |
| GitHub Actions | 作为 workflow artifact 上传 |

### 7.2 输出文件内容

结果 Excel 目前包含四个工作表：

#### `搜索结果`

- 按 BOM 原表风格导出
- 每组先父项，再子项
- 子项序号列留空
- `总数量` 会按输入数量与原 BOM 数量关系重算

#### `汇总BOM`

- 面向“旧 BOM 汇总生成新 BOM”场景
- 对结果中的有效子项去重汇总
- 若某组没有子项，则回退汇总父项本身
- `总数量` 累加，便于作为新 BOM 草稿继续整理

#### `搜索明细`

- 用于追踪来源文件、来源工作表、父子项行号等明细
- 便于排错和人工核对

#### `未找到`

- 记录未匹配到父图号的输入项

## 8. 常见失败排查

## 8.1 缺少 openpyxl

现象：运行时报 `ModuleNotFoundError: openpyxl`

处理方式：

```powershell
python -m pip install -r requirements.txt
```

## 8.2 缺少 tkinter，GUI 无法启动

现象：

- 运行 `python bom_searcher.py` 时 GUI 无法弹出
- 或报错提示当前环境不可用图形界面

处理方式：

1. 优先改用 CLI 模式：
   ```powershell
   python bom_searcher.py --input "..." --bom-folder "..." --output "..."
   ```
2. 如果必须使用 GUI，重新安装带 tkinter 的官方 Windows Python，或在系统中补齐对应 Tcl/Tk 组件

> 当前程序已做兼容：即使缺少 tkinter，只要使用 CLI 参数运行，CLI 仍可正常工作。

## 8.3 中文日志乱码

现象：控制台中中文出现乱码

处理方式：

1. 优先使用仓库脚本：
   ```powershell
   run_bom_searcher.bat
   verify_windows.bat
   ```
2. 这两个批处理已内置：
   ```cmd
   chcp 65001
   ```
3. 如仍乱码，建议改用 Windows Terminal / PowerShell 7，或确认终端编码为 UTF-8

## 8.4 输出文件被再次扫描，导致结果异常增多

现象：同一个目录重复运行后，结果越来越多或被历史结果污染

原因：输出文件被放入待扫描 BOM 目录，后续又参与了索引

规避方式：

1. **优先把输出文件写到单独目录**，例如：`artifacts\`
2. 若必须放在 BOM 目录中，建议文件名：
   - 包含 `搜索结果`
   - 或以下划线 `_` 开头
   - 或以 `result` / `output` 开头
3. 当前程序已自动跳过以下类型文件：
   - 名称包含 `搜索结果`
   - 文件名以下划线 `_` 开头
   - 文件名以 `result` 开头
   - 文件名以 `output` 开头

## 8.5 `.xls` 文件无法读取

现象：

- `.xlsx` 可正常运行，但 `.xls` 被跳过
- 日志中提示缺少 Excel COM 支持

原因：

- `.xls` 兼容路径依赖 Windows 本机 Excel COM
- 当前机器没有安装桌面版 Excel
- 或安装了 Excel，但 COM 不可用
- 或 `pywin32` 未正常安装

处理方式：

1. 确认是 Windows 环境
2. 安装桌面版 Excel
3. 确认 `pywin32` 已安装：
   ```powershell
   python -c "import win32com.client; print('pywin32 ok')"
   ```
4. 若交付允许，建议先将 `.xls` 统一转换为 `.xlsx` 再处理

> 注意：GitHub Actions 托管的 `windows-latest` 一般没有桌面版 Excel，因此不能把线上 CI 通过视为 `.xls` 场景也完全通过。

## 8.6 输入文件无有效图号

现象：程序报错提示“输入 Excel 里没有识别到图号列或有效图号”

处理方式：

- 检查待查 Excel 是否确实包含 `图号` 列
- 检查表头名称是否被合并、空格污染或写法异常
- 确认输入表中确实存在有效图号值

## 8.7 BOM 文件夹没有可搜索文件

现象：程序提示 BOM 文件夹下没有可搜索的 Excel 文件

处理方式：

- 检查目标目录中是否存在 `.xlsx/.xlsm/.xltx/.xltm/.xls`
- 检查是否把结果文件目录误传给了 `--bom-folder`
- 检查文件是否都被命名成被自动忽略的结果文件模式

## 9. CI 使用说明

仓库已提供最小 GitHub Actions 工作流：

```text
.github/workflows/windows-verify.yml
```

其作用是：

- 在 Windows Runner 上安装依赖
- 运行单元测试
- 执行一次 CLI 冒烟验证
- 上传验证结果作为 artifact

### 9.1 CI 的覆盖边界

当前 CI 能稳定覆盖：

- `.xlsx`
- `.xlsm`
- `.xltx`
- `.xltm`
- 核心 Python 搜索与汇总逻辑
- CLI 可运行性

当前 CI **不能完整替代** 的部分：

- `.xls + Excel COM` 路径
- 真正的桌面 GUI 人工交互路径
- 客户现场 Excel 安装状态差异

如需覆盖 `.xls` 真实路径，建议使用：

- 自托管 Windows Runner
- 或线下验收机 / 客户现场机器
