## 2. 当前阶段性成果与验收边界

### 2.1 当前已具备的能力
基于当前实现与测试，可确认系统已经具备：
- 从任务单 Excel 读取 `序号 / 图号 / 名称 / 数量`
- 从旧BOM目录递归扫描 Excel 文件
- 按图号主编码匹配旧BOM父项
- 提取父项下的子项、非标准子序号行、空序号说明行
- 按任务单数量对父项/子项 `总数量` 重算
- 导出四张表：
  - `搜索结果`
  - `汇总BOM`
  - `搜索明细`
  - `未找到`
- 当前自动化验证状态：
  - 单元测试 `6` 项通过
  - Windows CI 已配置
  - CLI 冒烟验证已落地并输出 `artifacts/verify_output.xlsx`

### 2.2 当前仍存在的业务偏差 / 待拍板项
以下问题会影响“正式业务验收是否通过”：
1. **说明行是否进入汇总BOM**
   - 业务建议：**不进入**
   - 当前实现：存在被纳入汇总的风险，属于 P0 待修正点
2. **同一父图号命中多个旧BOM来源时如何处理**
   - 业务建议：不能默认全部累加，至少要支持“待确认/可配置策略”
   - 当前实现：会全部进入结果
3. **聚合是否按图号主编码还是完整图号**
   - 业务建议：需根据版本管理口径明确
   - 当前实现：默认按 `drawing_base` 聚合

> 结论：**当前需求已基本完整，但仍非100%完成定义。**
> 最需要尽快拍板的，是“说明行是否入汇总”“多来源重复命中策略”“完整图号/主图号聚合口径”。

## 3. 最终验收时建议检查的输出对象

建议最终验收至少检查 4 类输出对象：

### 3.1 `汇总BOM`（主结果）
面向采购、生产、计划人员使用，是“新BOM草稿/主结果”。

### 3.2 `展开明细` / `搜索明细`
面向核对与追溯，保留：
- 任务单输入行
- 旧BOM来源文件/工作表
- 命中的父项
- 提取到的子项
- 重算数量

### 3.3 `未找到`
记录完全找不到旧BOM父项的输入图号。

### 3.4 `异常报告`（建议最终版本新增）
记录：
- 多版本冲突
- 数量无法可靠换算
- 字段冲突
- 说明行被过滤/待确认

> 当前实现已稳定输出前 3 类结果和 `汇总BOM`，但**异常报告**仍建议作为最终验收版本补齐。

## 4. 典型任务单输入示例

## 4.1 示例A：两个总成，存在共用子件

### 任务单输入
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 10 | 100-A | 总成A | 2 |
| 20 | 100-B | 总成B | 3 |

### 业务意义
- 同一个任务单里有两个父项总成
- 两个父项可能展开出同一个共用子件
- 新BOM需要把共用子件自动汇总

---

## 4.2 示例B：同一父图号在任务单中重复出现

### 任务单输入
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 84 | 101002000250(248010310) | 支架 | 18 |
| 85 | 101002000250(248010310) | 支架 | 6 |

### 业务意义
- 同一个父图号在不同任务行重复出现
- 用户期望：
  - 明细中保留两条来源
  - 汇总BOM中相同子件数量自动累加

---

## 4.3 示例C：找到父项，但没有任何子项

### 任务单输入
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 1 | SOLO-100 | 单独总成 | 5 |

### 业务意义
- 某些旧BOM只有父项本身，没有子件结构
- 用户仍希望该父项以“父项兜底”的方式进入新BOM

---

## 4.4 示例D：任务单中存在无法匹配的图号

### 任务单输入
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 1 | 999999999999 | 不存在的总成 | 1 |

### 业务意义
- 用户需要系统明确告诉他“哪个图号没找到”，而不是静默忽略

|  |  | 工艺说明 |  |  |  |  | 同组备注 |

## 5. 旧BOM来源组合方式定义

业务上建议把旧BOM来源组合分成 4 种情况理解：

## 5.1 单父项 -> 单旧BOM来源
最简单场景。
- 一个输入图号
- 命中一个旧BOM文件中的一个父项
- 展开得到一组子项

**验收期望**：
- 匹配准确
- 子项数量重算准确
- 可追溯到来源文件/工作表/行号

## 5.2 多父项 -> 多旧BOM来源
例如：
- 任务单中有 A、B 两个总成
- A 命中 `bom_a.xlsx`
- B 命中 `bom_b.xlsx`
- 两边展开后汇总成一个新BOM

**验收期望**：
- 不同旧BOM来源可以共同组成一个新BOM
- 共用子件应被合并

## 5.3 同一父项在任务单中多次出现
例如：
- 图号 `101002000250` 在任务单出现 2 次
- 数量分别为 `18` 与 `6`

**验收期望**：
- 展开明细中保留两次来源
- 新BOM汇总中同一子件数量按两次重算结果相加

## 5.4 同一父图号在多个旧BOM中重复命中
例如：
- `100-A` 同时命中 `bom_a_v1.xlsx` 和 `bom_a_v2.xlsx`

**业务验收期望**：
- 不能默认把两个版本都算进去
- 应符合以下之一：
  1. 自动选定一个优先版本
  2. 进入异常/待确认
  3. 允许用户配置 `all-matches / first-match / latest-file / manual-review`

**当前实现现状**：
- 会全部纳入结果
- 正式业务验收时，此项应视为**需确认口径或待修正项**

## 6. 新BOM主输出字段定义

## 6.1 当前最小可交付字段（已落地）
当前代码与结果文件中的 `汇总BOM` 使用以下列：

| 字段 | 是否必需 | 说明 |
|---|---|---|
| 序号 | 否 | 当前固定留空，建议最终版本重排生成 |
| 物料ID | 建议必需 | 同料去重的最高优先标识 |
| 图号 | 必需 | 物料图号/零件图号 |
| 名称 | 必需 | 物料名称 |
| 厚度 | 可空 | 区分规格 |
| 材质 | 可空 | 区分规格 |
| 数量 | 可空 | 若同一聚合项出现多个不同值，建议留空 |
| 总数量 | 必需 | 新BOM真正关心的汇总数量 |
| 备注 | 可空 | 备注、工艺提示或异常提示 |

## 6.2 最终业务验收建议字段（推荐版）
为支撑最终评审与生产使用，建议在 `汇总BOM` 或附加表中补齐：

| 字段 | 来源 | 说明 |
|---|---|---|
| 序号 | 汇总生成 | 不使用旧BOM层级序号 |
| 物料ID | 子项优先 / 父项兜底 | 汇总一级键 |
| 图号 | 子项优先 / 父项兜底 | 汇总一级或二级键 |
| 名称 | 子项优先 / 父项兜底 | 展示字段 |
| 厚度 | 子项 | 规格拆分 |
| 材质 | 子项 | 规格拆分 |
| 单件数量 | 子项 `quantity` | 仅在值一致时展示 |
| 汇总总数量 | 重算后累加 | 核心业务数量 |
| 备注 | 备注 + 异常提示 | 供人工复核 |
| 来源父图号集合 | Query + Parent | 追溯来源 |
| 来源任务单序号集合 | Query.sequence | 追溯来源 |
| 来源文件集合 | 文件名集合 | 追溯来源 |
| 匹配状态 | 汇总状态机 | 如 `AGGREGATED / CONFLICT_SPLIT / QTY_UNRESOLVED` |

## 7. 字段优先级定义

## 7.1 主识别字段优先级
建议业务上按以下优先级识别同一物料：
1. `物料ID`
2. `完整图号`
3. `图号主编码(drawing_base)`
4. `图号 + 名称 + 厚度 + 材质`

> 当前实现实际聚合键为：
> `物料ID + drawing_base + 名称 + 厚度 + 材质 + 备注`
>
> 这可以作为首版交付口径，但若业务强调版本号/括号尾码隔离，最终验收前需确认是否改为“完整图号聚合”。

## 7.2 字段来源优先级
| 字段 | 优先级规则 |
|---|---|
| 物料ID | 子项优先；无子项时父项兜底 |
| 图号 | 子项优先；无子项时父项兜底 |
| 名称 | 子项优先；无子项时父项兜底 |
| 厚度 | 子项值优先；无值可空 |
| 材质 | 子项值优先；无值可空 |
| 数量 | 若同聚合项只有一个值则展示；多个不同值则清空 |
| 总数量 | 永远以“重算后累计值”为准 |
| 备注 | 同值保留一次；不同值建议去重拼接或写异常说明 |

## 7.3 数量字段优先级
1. **汇总总数量**：以重算累计结果为准
2. **原总数量**：仅作为换算基数，不作为最终汇总值
3. **原数量**：用于降级计算与参考展示

## 8. 去重/合并示例（核心验收样例）

## 8.1 示例1：共用子件自动合并

### 输入任务单
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 10 | 100-A | 总成A | 2 |
| 20 | 100-B | 总成B | 3 |

### 旧BOM A
| 序号 | 物料ID | 图号 | 名称 | 厚度 | 材质 | 数量 | 总数量 | 备注 |
|---|---|---|---|---|---|---|---|---|
| 1 | PARENT-A | 100-A | 总成A |  |  | 1 | 1 |  |
| 1.1 | CH-001 | PART-X | 子件X | T2 | SUS304 | 2 | 2 | 共用 |
| 1.2 | CH-002 | PART-Y | 子件Y | T1 | Q235 | 1 | 1 |  |

### 旧BOM B
| 序号 | 物料ID | 图号 | 名称 | 厚度 | 材质 | 数量 | 总数量 | 备注 |
|---|---|---|---|---|---|---|---|---|
| 1 | PARENT-B | 100-B | 总成B |  |  | 1 | 1 |  |
| 1.1 | CH-001 | PART-X | 子件X | T2 | SUS304 | 2 | 2 | 共用 |
| 1.2 | CH-003 | PART-Z | 子件Z | T3 | AL | 4 | 4 |  |

### 期望汇总BOM
| 物料ID | 图号 | 名称 | 厚度 | 材质 | 数量 | 总数量 | 备注 |
|---|---|---|---|---|---|---|---|
| CH-001 | PART-X | 子件X | T2 | SUS304 | 2 | 10 | 共用 |
| CH-002 | PART-Y | 子件Y | T1 | Q235 | 1 | 2 |  |
| CH-003 | PART-Z | 子件Z | T3 | AL | 4 | 12 |  |

### 计算说明
- `PART-X`
  - 来源A：`2 × 2 / 1 = 4`
  - 来源B：`2 × 3 / 1 = 6`
  - 汇总：`4 + 6 = 10`
- `PART-Y`
  - 来源A：`1 × 2 / 1 = 2`
- `PART-Z`
  - 来源B：`4 × 3 / 1 = 12`

### 验收重点
- 共用子件只保留一行
- 总数量计算准确
- 备注 `共用` 被保留

## 8.4 示例4：说明行和非标准子序号行的处理

### 输入任务单
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 55 | 101002000878 | 手柄总成 | 9 |

### 期望明细行为
- `A / 101002009999 / 无1.1格式零件` 作为有效子项进入明细
- `工艺说明 / 同组备注` 作为说明行进入明细

### 期望汇总BOM行为
- `101002009999` 应进入 `汇总BOM`
- `工艺说明` 这类无图号/无物料ID说明行，**不应进入汇总BOM**

### 验收重点
- 系统不能只识别 `1.1` 样式的子项
- 说明行不能被误当成物料汇总

> 备注：当前实现对“说明行不进入汇总BOM”尚需重点验证或修正，此项建议列为最终业务验收必查项。

## 8.5 示例5：完全未命中的处理

### 输入任务单
| 序号 | 图号 | 名称 | 数量 |
|---|---|---|---|
| 1 | 999999999999 | 不存在的总成 | 1 |

### 期望输出
- `汇总BOM`：不生成该物料
- `未找到`：出现一行，标记 `未找到匹配图号`
- `搜索明细`：不伪造任何匹配记录

### 验收重点
- 未命中信息必须明确、可定位、可补录

## 8.6 示例6：同图号多来源重复命中

### 场景
同一输入图号 `100-A` 同时在以下文件中命中：
- `bom_a_v1.xlsx`
- `bom_a_v2.xlsx`

### 业务期望
必须采用明确策略之一：
1. 只取一个最优来源
2. 标记为待确认
3. 由用户显式选择

### 不建议接受的结果
- 两个版本都自动合并进 `汇总BOM`，且没有任何提示

### 验收重点
- 系统不能“静默重复累加”
- 必须能让用户知道发生了多来源命中

## 9. 异常与边界案例清单

| 场景 | 期望系统行为 | 是否应进入汇总BOM | 是否应进入异常/未找到 |
|---|---|---|---|
| 输入图号为空 | 跳过，不匹配 | 否 | 可记录 |
| 输入数量为空 | 允许查询，但数量标记待确认 | 视规则而定 | 建议是 |
| 输入数量非数字 | 允许查询，但不应静默算0 | 视规则而定 | 建议是 |
| 未找到旧BOM | 不生成物料 | 否 | 是，进入 `未找到` |
| 有父项无子项 | 父项兜底 | 是 | 否 |
| 说明行 | 明细保留 | 否 | 否/可做提示 |
| 非标准子序号行 | 当有效子项处理 | 是 | 否 |
| 同一父图号多来源命中 | 待确认或按策略取一份 | 不应默认全部进 | 是，建议异常 |
| 同聚合项 `数量` 不一致 | `数量` 列置空，`总数量` 继续累计 | 是 | 建议提示 |
| `.xls` 无法转换 | 跳过文件并记录 | 与该文件相关部分否 | 是，建议日志/异常 |

## 10. 用户视角下“验收通过”的定义

从用户角度，以下结果同时满足时，可判定“新BOM汇总”业务验收通过：

### 10.1 可用性通过
1. 用户只需提供：
   - 一个任务单 Excel
   - 一个旧BOM目录
   - 一个输出路径
2. 系统能稳定生成结果文件
3. 输出文件至少包含：
   - `汇总BOM`
   - `搜索明细`
   - `未找到`

### 10.2 业务正确性通过
1. 父项匹配正确，不把不相干图号匹配上
2. 子项展开边界正确，不漏掉同组有效子项
3. 总数量重算正确
4. 相同物料在汇总BOM中被正确合并
5. 同一输入图号重复出现时，汇总数量能够累计
6. 无子项时能回退保留父项
7. 未找到项被单独列出，不静默丢失

### 10.3 可追溯性通过
1. 用户能从汇总BOM追到来源父图号、来源文件、来源任务单序号
2. 用户能从明细表反查每条物料是从哪个旧BOM来的
3. 用户能定位未找到或异常项并人工处理

### 10.4 风险控制通过
1. 说明行不应被当成标准物料进入汇总BOM
2. 多版本旧BOM重复命中不能无提示地自动累加
3. 图号版本差异是否合并必须有明确规则

## 11. 最终验收建议清单（Review Checklist）

建议评审时按以下清单逐项核对：

## 12. 建议的业务验收标准（Gherkin）

```gherkin
Feature: 任务单汇总生成新BOM

  Scenario: 多个总成展开后生成统一汇总BOM
    Given 任务单中包含多个父项图号和数量
    And 旧BOM目录中存在对应父项及子项结构
    When 用户执行新BOM汇总
    Then 系统应输出汇总BOM、搜索明细和未找到结果
    And 相同子件应在汇总BOM中自动合并
    And 汇总总数量应等于各来源重算数量之和

  Scenario: 同一父图号在任务单中重复出现
    Given 任务单中同一父图号出现多次且数量不同
    When 系统执行汇总
    Then 搜索明细中应保留各输入来源
    And 汇总BOM中相同子件应合并为一行并累加数量

  Scenario: 旧BOM中存在说明行
    Given 旧BOM父项下存在无图号的工艺说明行
    When 系统执行汇总
    Then 说明行应保留在搜索明细中
    And 说明行不应进入汇总BOM

  Scenario: 某输入图号未命中任何旧BOM
    Given 任务单中存在未命中的图号
    When 系统执行汇总
    Then 该图号应进入未找到工作表
    And 汇总BOM中不应出现伪造结果

  Scenario: 同一父图号命中多个旧BOM来源
    Given 输入图号在多个旧BOM文件中均有候选父项
    When 系统无法自动确定唯一来源
    Then 系统不应静默把多个版本全部合并进主汇总结果
    And 系统应给出异常提示、待确认标记或采用已配置策略
```
