# Tkinter 界面改造草案（面向 `bom_searcher.py` 落地）

## 1. 文档目的

本文是在前期“前端评估”和“实施清单与字段映射”的基础上，进一步整理成一份可执行的 **Tkinter 界面改造草案**。

目标不是重新设计一套抽象 UI，而是：
- 尽量对齐当前 `bom_searcher.py` 的结构；
- 说明窗口应该如何重新布局；
- 列出建议新增的控件、变量、校验和交互；
- 串起用户从“选择任务单”到“生成新 BOM 并导出”的完整流程；
- 让后续开发可以直接按此草案改造 `BOMSearcherApp`。

---

## 2. 当前代码现状与改造原则

## 2.1 当前 GUI 现状

当前图形界面在：
- `bom_searcher.py:851-998`（`BOMSearcherApp`）

现有界面结构非常简单：
1. 标题和说明
2. 三行路径输入：待查 Excel / BOM 文件夹 / 输出结果
3. 两个按钮：开始搜索 / 清空日志
4. 一个文本日志框

这套结构适合“第一版搜索工具”，但已经不适合当前需求：
- 目前代码已经新增了 `汇总BOM` 工作表导出能力（`write_aggregated_sheet()`）；
- 但 GUI 没有对应的模式切换、结果预览、导出控制、错误列表和校验提示；
- 用户仍然只能“盲跑”一次，再自己打开 Excel 看结果。

## 2.2 改造原则

### 原则 A：先增强 `BOMSearcherApp`，不强制立即拆文件
短期内建议仍在 `bom_searcher.py` 内完成第一阶段增强，优先确保能跑。

### 原则 B：保留现有主流程，先让 GUI 能承接新流程
优先复用：
- `collect_queries()`
- `build_index()`
- `search_queries()`
- `write_output()`
- `run_search()`

同时在 GUI 层新增：
- 运行模式
- 输入校验
- 结果预览
- 异常/空结果提示

### 原则 C：布局要让用户“先确认输入，再执行”
不能继续保持“选完路径就立即运行”的黑盒体验，应增加：
- 工作表选择
- 字段识别/字段确认
- 任务单预览
- 执行状态提示
- 结果页签

## 3. 推荐窗口布局

建议将当前单列布局改造成 **上中下三层 + 左右双栏**。

## 3.1 顶部区域：标题与模式切换

### 位置
窗口最上方，占整行。

### 内容
- 标题：`BOM 搜索与汇总工具`
- 副标题：`从任务单中读取图号/数量，到历史 BOM 库中搜索、展开并汇总生成新 BOM。`
- 运行模式：
  - `搜索模式`
  - `新BOM汇总模式`

### 对应建议控件
- `title_label`
- `desc_label`
- `run_mode_combo`

### 推荐变量
```python
self.run_mode_var = self.tk.StringVar(value="summary")
```

### 交互行为
- `search` 模式：
  - 保留现有逻辑为主；
  - 可以简化隐藏部分汇总选项；
  - 导出内容以 `搜索结果/汇总BOM/搜索明细/未找到` 为主。
- `summary` 模式：
  - 显示全部字段映射和高级选项；
  - 强制进行任务单列映射与校验。

---

## 3.2 中部区域：左右双栏主工作区

### 左侧：配置区
建议宽度约 34%~38%，垂直分 4 个分组框：
1. 输入设置
2. 字段映射
3. 汇总/导出配置
4. 操作按钮

### 右侧：预览与结果区
建议宽度约 62%~66%，使用 `ttk.Notebook`。

---

## 3.3 底部区域：状态栏

### 内容
- 当前状态：待执行 / 校验通过 / 正在扫描 / 已完成 / 失败
- 扫描进度提示（文本形式即可）
- 执行摘要：命中多少、未找到多少、跳过文件多少
- 当前输出路径

### 推荐变量
```python
self.status_var = self.tk.StringVar(value="待执行")
self.summary_hint_var = self.tk.StringVar(value="")
self.output_hint_var = self.tk.StringVar(value="")
```

## 4. 左侧配置区草案

## 4.1 分组一：输入设置

### 目标
让用户完成执行前最基本的来源与输出配置。

### 控件清单

#### 1）任务单文件
- 控件名：`task_file_entry`
- 变量：`task_file_var`
- 类型：`StringVar`
- 按钮：`task_file_button`
- 事件：`choose_task_file()`

**用途**
选择任务单 Excel。

**校验**
- 必填
- 文件必须存在
- 扩展名必须为：`.xlsx/.xlsm/.xltx/.xltm/.xls`

**失败提示**
- `请选择任务单 Excel 文件。`
- `任务单文件不存在，请重新选择。`
- `任务单文件格式不支持，仅支持 Excel 文件。`

#### 2）任务单工作表
- 控件名：`task_sheet_combo`
- 变量：`task_sheet_var`
- 类型：`StringVar`
- 类型控件：`ttk.Combobox`
- 状态：`readonly`

**用途**
在多工作表任务单中指定处理的工作表。

**交互**
- 选择任务单文件后自动加载工作表列表；
- 若只有一个表，自动选中；
- 若多个表，提示用户选择。

**失败提示**
- `请选择要处理的任务单工作表。`

#### 3）BOM 文件夹
- 控件名：`bom_folder_entry`
- 变量：`bom_folder_var`
- 按钮：`bom_folder_button`
- 事件：`choose_bom_folder()`

**用途**
指定历史旧 BOM 所在目录。

**校验**
- 必填
- 目录必须存在

**失败提示**
- `请选择 BOM 文件夹。`
- `BOM 文件夹不存在，请重新选择。`

#### 4）输出结果文件
- 控件名：`output_file_entry`
- 变量：`output_file_var`
- 按钮：`output_file_button`
- 事件：`choose_output_file()`

**用途**
指定结果文件保存位置。

**建议默认命名**
- 搜索模式：`任务单文件名_搜索结果.xlsx`
- 汇总模式：`任务单文件名_汇总BOM.xlsx`

**校验**
- 必填
- 后缀必须为 `.xlsx`

**失败提示**
- `请选择结果输出路径。`
- `输出文件必须为 .xlsx 格式。`

#### 5）任务单预览按钮
- 控件名：`preview_task_button`
- 方法：`_preview_task_items()`

**用途**
在真正执行前，先预览任务单中将被当成查询项的内容。

每个业务字段一行：
- 左：字段名称 Label

### 控件清单

- 控件名：`task_no_combo`
- 变量：`task_no_col_var`
- 必填性：可选
- 用途：用于追溯、输出命名、后续扩展按任务单号分组

#### 2）输入序号列
- 控件名：`sequence_combo`
- 变量：`sequence_col_var`
- 必填性：建议有
- 用途：生成来源任务序号集合

#### 3）图号列
- 控件名：`drawing_combo`
- 变量：`drawing_col_var`
- 必填性：必填
- 用途：构建 `QueryItem.drawing` 和 `drawing_base`

**失败提示**

#### 4）名称列
- 控件名：`name_combo`
- 变量：`name_col_var`
- 必填性：建议有

#### 5）数量列
- 控件名：`quantity_combo`
- 变量：`quantity_col_var`
- 必填性：汇总模式必填

#### 6）备注列
- 控件名：`remark_combo`
- 变量：`remark_col_var`

- 用途：保留任务单备注

#### 7）识别字段按钮
- 控件名：`detect_mapping_button`
- 方法：`_detect_task_columns()`

**用途**

## 4.3 分组三：汇总/导出配置

### 目标
把“现在代码里写死”的行为逐步转成用户可理解、可选择的配置。

### 建议控件

#### A. 匹配模式
- 控件名：`match_mode_combo`
- 变量：`match_mode_var`
- 选项：
  - `merge_all`：全部保留并汇总
  - `strict_best`：仅取优先级最高来源

**用途**
处理同一图号命中多个旧 BOM 时的策略。

#### B. 汇总主键
- 控件名：`dedup_key_combo`
- 变量：`dedup_key_var`
- 选项：
  - `material_id_first`
  - `drawing_first`

**用途**
控制新 BOM 汇总去重逻辑。

#### C. 是否保留父项行
- 控件名：`include_parent_check`
- 变量：`include_parent_var`
- 类型：`BooleanVar`

**用途**
决定汇总结果里是否保留父项总成行。

#### D. 是否保留说明行到明细
- 控件名：`include_note_check`
- 变量：`include_note_var`

**用途**
控制空序号说明行是否只进明细、不进汇总。

#### E. 是否生成异常报告
- 控件名：`generate_exception_check`
- 变量：`generate_exception_var`

#### F. 导出内容复选框
- `export_summary_check` -> `export_summary_var`
- `export_detail_check` -> `export_detail_var`
- `export_not_found_check` -> `export_not_found_var`
- `export_exception_check` -> `export_exception_var`

**校验**
至少选择一个导出内容。

**失败提示**
- `请至少选择一个导出内容。`

#### G. 导出后自动打开
- 控件名：`open_after_export_check`
- 变量：`open_after_export_var`

## 6. 输入校验与提示方式草案

## 6.1 校验分层

### 第一层：路径级校验（点击“开始执行”前）
- 任务单文件是否存在
- BOM 目录是否存在
- 输出路径是否为空

### 第二层：结构级校验（字段识别/预览时）
- 是否识别到图号列
- 汇总模式下是否识别到数量列
- 工作表是否为空
- 是否有有效输入项

### 第三层：执行结果校验（运行完成后）
- 是否存在未找到
- 是否存在跳过文件
- 是否存在多来源冲突
- 是否存在数量异常

## 6.2 提示方式推荐

### A. `messagebox.showwarning`
用于执行前阻断型问题，例如：
- `请选择任务单 Excel 文件。`
- `任务单中未识别到“图号”列，请手动指定。`
- `汇总模式必须指定“数量”列。`

### B. 状态栏提示
用于轻量进度反馈，例如：
- `已加载任务单，等待字段识别。`
- `字段识别完成，请确认图号和数量列。`
- `正在扫描 BOM 文件...`
- `执行完成，但存在异常，请查看“异常报告”。`

### C. 日志框
用于详细过程信息，例如：
- `正在读取待查 Excel...`
- `待扫描 BOM 文件数: 126`
- `已扫描: 260118.xlsx，索引到 1860 条候选图号`
- `跳过文件: xxx.xls，原因: 当前环境缺少 Excel COM 支持`

### D. 结果页签提示
用于可视化承载结果，例如：
- 未找到 -> `not_found_tree`
- 冲突/跳过 -> `exception_tree`
- 预览异常 -> `task_preview_tree` 行样式

## 7. 用户完整交互流程草案

下面是建议的用户主流程，开发时应尽量让每一步都有对应的 UI 响应。

## 7.1 步骤 1：启动窗口

### 用户看到
- 标题
- 模式切换
- 左侧配置区
- 右侧空白结果页签
- 底部状态：`待执行`

### 系统行为
- 默认模式为 `新BOM汇总模式`
- `run_button` 默认可用
- 结果页签为空表

---

## 7.2 步骤 2：选择任务单

### 用户操作
点击“选择任务单”。

### 系统行为
- 打开文件选择框
- `task_file_var` 赋值
- 自动读取工作簿名称列表
- 填充 `task_sheet_combo`
- 若 `output_file_var` 为空，则自动生成输出文件名
- 状态栏更新：`已加载任务单，等待字段识别。`

### 对应方法建议
- 现有 `choose_input_file()` 改名/扩展为 `choose_task_file()`
- 新增 `_load_task_sheets()`

---

## 7.3 步骤 3：识别字段

### 用户操作
点击“识别字段”。

### 系统行为
- 读取任务单工作表前若干行
- 自动识别表头
- 自动填充：序号 / 图号 / 名称 / 数量 / 备注
- 对未识别到的关键列显示警告
- 状态栏更新：
  - `字段识别完成，请确认映射。`
  - 或 `未识别到图号列，请手动指定。`

### 对应方法建议
- 新增 `_detect_task_columns()`
- 底层复用 `detect_header()` 和 `HEADER_ALIASES`

---

## 7.4 步骤 4：预览任务单

### 用户操作
点击“预览任务单”。

### 系统行为
- 根据当前映射抽取预览数据
- 填充 `任务单预览` 页签
- 标出异常行：
  - 图号为空
  - 数量为空
  - 疑似重复图号
- 如果没有有效输入项：
  - 阻止执行
  - 提示：`任务单中没有识别到有效图号，请检查字段映射。`

### 对应方法建议
- 新增 `_preview_task_items()`

## 7.8 步骤 8：执行完成

  - `异常报告` 页签，或
  - `未找到` 页签
- 弹窗文案应更温和：
  - `执行已完成，但存在未找到或冲突数据，请先核对结果后再使用。`

- 恢复按钮可点击

- 保留 `_on_search_success()`
- 保留 `_on_search_failed()`

  - 填充 `detail_tree`
  - 填充 `not_found_tree`
  - 填充 `exception_tree`
  - 更新 `status_var` / `summary_hint_var`

## 8. 空结果、冲突数据、错误提示草案

## 8.1 空结果

### 场景 A：输入为空
- 提示：`任务单中没有识别到有效图号，请检查字段映射。`
- 类型：阻断执行

### 场景 B：BOM 文件夹为空
- 提示：`BOM 文件夹下没有可搜索的 Excel 文件。`
- 类型：阻断执行

### 场景 C：全部未命中
- 提示：`执行完成，但没有匹配到任何旧 BOM，请查看“未找到”页签。`
- 类型：允许导出未找到表

---

## 8.2 冲突数据

### 场景 A：同一图号命中多个旧 BOM
- 日志：详细列出来源文件
- 状态栏：`存在多来源命中，已按当前匹配策略处理。`
- 异常报告：记录一条或多条冲突项

### 场景 B：数量无法换算
- 日志：`图号 XXX 的部分子项数量无法换算，已写入异常报告。`
- 页签：进入 `异常报告`

### 场景 C：字段冲突
例如相同图号但材质/厚度不同：
- 页签中标记 `CONFLICT_SPLIT`
- 汇总BOM允许拆成多行

---

- `.xls` 无法转换

- 工作簿读取失败

### 提示方式
- 日志逐条记录
- 异常报告页签归档
- 完成弹窗提示数量：`跳过文件 3 个`

## 11. 最终建议

对当前项目而言，Tkinter 界面改造的关键不是“换个好看点的布局”，而是把 GUI 从一个简单的“参数输入器”升级成一个 **可预览、可校验、可解释结果的新BOM汇总工作台**。

如果只做最小可行版本，建议优先完成以下 5 件事：
1. 窗口改成左右分栏 + 结果页签；
2. 增加任务单工作表选择和字段识别；
3. 增加任务单预览表；
4. 执行完成后在 GUI 中显示 `汇总BOM / 展开明细 / 未找到`；
5. 加入状态栏和更明确的错误/异常提示。

这样即使后端汇总逻辑后续继续增强，GUI 也已经具备了承接“任务单 -> 搜索 -> 汇总 -> 导出”的完整交互骨架。
